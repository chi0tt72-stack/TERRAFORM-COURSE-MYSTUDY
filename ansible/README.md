# Ansible in this project

Ansible is used here for **configuration management** of the EC2 instances that Terraform creates. Terraform provisions the infrastructure; Ansible configures the OS and software on those instances.

## Flow

1. **Terraform** creates VPC, subnets, EC2 instances, security groups, etc.
2. **Terraform** writes an Ansible inventory file from its outputs (`ansible/inventory/terraform_hosts.ini`).
3. **You** run Ansible playbooks against that inventory to install packages, deploy configs, and manage the instances.

## Prerequisites

- Python 3 and [Ansible](https://docs.ansible.com/ansible/latest/installation_guide/index.html) installed locally (e.g. `pip install ansible` or your OS package manager).
- Terraform already applied so EC2 instances exist and `ansible/inventory/terraform_hosts.ini` has been generated.
- SSH access to the instances using the same key Terraform uses (e.g. the key whose public part is in AWS Secrets Manager `terraform/ssh-public-key`). Typically that key is at `~/.ssh/terraform-course-key` (or set `ansible_ssh_private_key_file`).

### Installing Ansible on Windows

- **Option A — WSL (recommended):** Install [WSL](https://docs.microsoft.com/en-us/windows/wsl/install), open an Ubuntu (or other) shell, then:
  ```bash
  sudo apt update && sudo apt install -y ansible
  ```
  Run all Ansible commands from that WSL shell (use the same paths under `/mnt/c/...` if your repo is on `C:\`).
- **Option B — Python on Windows:** Install [Python 3](https://www.python.org/downloads/) (ensure "Add Python to PATH" is checked), open a new PowerShell, then:
  ```powershell
  pip install ansible
  ```
  If `ansible-playbook` is still not found, use: `python -m ansible.playbook playbooks/site.yml ...` from the `ansible` directory.

## Quick start

From the **repository root**:

```bash
# 1. Apply Terraform (creates/updates instances and writes Ansible inventory)
cd environments/dev
terraform apply
cd ../..

# 2. Run Ansible (use the same SSH key you use for EC2)
cd ansible
ansible-playbook playbooks/site.yml -e ansible_ssh_private_key_file=~/.ssh/terraform-course-key
```

On **PowerShell**, use the full path or `$env:USERPROFILE` so the key path is expanded:
```powershell
cd ansible
ansible-playbook playbooks/site.yml -e "ansible_ssh_private_key_file=$env:USERPROFILE\.ssh\terraform-course-key"
```

Or set the key once in `group_vars/all.yml` (uncomment and set `ansible_ssh_private_key_file`), then:

```bash
cd ansible
ansible-playbook playbooks/site.yml
```

## Layout

| Path | Purpose |
|------|--------|
| `ansible.cfg` | Ansible defaults; points to `inventory/terraform_hosts.ini`. |
| `inventory/terraform_hosts.ini` | **Generated by Terraform** from EC2 outputs (do not edit by hand). |
| `inventory/terraform_hosts.ini.example` | Example inventory if you want to maintain one manually. |
| `group_vars/all.yml` | Shared vars (e.g. `ansible_user`, optional SSH key path). |
| `playbooks/site.yml` | Main playbook: base packages, motd, etc. |

## What the sample playbook does

- Waits for SSH on the instance.
- Installs base packages (e.g. `python3`, `vim-enhanced`, `htop`, `tmux`) on Amazon Linux.
- Writes a simple `/etc/motd`.

You can add more playbooks or roles under `playbooks/` and `roles/` for apps, users, cron jobs, etc.

## Optional: run Ansible in CI after Terraform apply

To run Ansible automatically after `terraform apply` in GitLab CI, add a job that:

1. Depends on the `apply` job.
2. Installs Ansible and SSH (or uses an image that has them).
3. Uses GitLab CI variables or OIDC for the SSH private key (store the key in a CI variable or Secrets Manager and write it to a temp file in the job).
4. Runs from the repo root: `cd ansible && ansible-playbook playbooks/site.yml -e ansible_ssh_private_key_file=$SSH_KEY_PATH`.

This is optional; many teams run Ansible only from a controlled workstation or a separate pipeline.
